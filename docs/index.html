The Big Picture
What are we doing, and why does it matter?

Imagine you work for the City of Austin and you want to answer questions like: "Which bike stations are most popular in the morning rush?" or "How has ride duration changed over the years?"

Google BigQuery has a public dataset with every single Austin bikeshare trip ever recorded. That's millions of rows of raw data. The problem is, raw data is messy — it's designed for record-keeping, not for answering business questions.

🍳 Analogy
Think of raw data as a fridge full of ingredients. You could eat raw flour and uncooked eggs, but what you really want is a finished meal. dbt (data build tool) is your recipe book — it transforms those raw ingredients into something delicious and useful.

⚠️ The Problem with Raw Data
Raw data changes over time. Stations open and close. What if you want to know what the station data looked like last year? Raw tables don't keep history — they just show the current state. We'll solve this in Step 5 with Snapshots.

Designing a Star Schema
The blueprint for analysis-ready data

Before we write any code, we need a plan. In data warehousing, we use a design pattern called a Star Schema.

🌟 Analogy
A star schema looks like a star in a night sky. In the center is a Fact table — it records events (things that happened). Surrounding it are Dimension tables — they describe the "who, what, where" context for those events.

For our Austin Bikeshare project, the design is:

Dimension
dim_station
start_station_id → station_id
↕
end_station_id → station_id
Fact
fact_trips

What goes in each table?

The fact table captures what happened — each row is one bike trip. It stores numbers you can measure (like duration) and foreign keys pointing to dimension tables.

fact table fact_trips
ColumnTypeRoleDescription
trip_idINTEGERPrimary KeyUnique ID for each trip
subscriber_typeSTRINGAttributeMembership type (Annual, Walk Up, etc.)
bike_idSTRINGAttributeWhich bicycle was used
start_timeTIMESTAMPMeasurementWhen the trip started
start_station_idINTEGERForeign Key→ links to dim_station
end_station_idINTEGERForeign Key→ links to dim_station
duration_minutesINTEGERMeasurementHow long the trip took ← the metric!

The dimension table captures the "who and where" — each row is one bike station. It stores descriptive attributes that give context to the trips.

dimension table dim_station
ColumnTypeRoleDescription
station_idINTEGERPrimary KeyUnique ID for each station
nameSTRINGAttributeHuman-readable station name
locationSTRINGAttributeGPS coordinates
addressSTRINGAttributeStreet address
property_typeSTRINGAttributeCity Owned or Private
number_of_docksINTEGERAttributeCapacity of the station
dbt_valid_fromTIMESTAMPSCD TrackWhen this version of the record started
dbt_valid_toTIMESTAMPSCD TrackWhen this version ended (NULL = current)
💡 Why separate tables?
Station names and addresses don't change with every trip. By keeping them in a separate dimension table, we avoid storing "Congress Ave & 2nd St" millions of times. We store it once and reference it by ID — cleaner, faster, and easier to update.

Setting Up the dbt Project
Your project folder structure and configuration

A dbt project is just a folder with a specific structure — SQL files, YAML configuration files, and a connection to your database. Think of it like a well-organized recipe binder.

Step 1 — Create the project

Run this command in your terminal. It creates the folder structure automatically:

terminal
dbt init austin_bikeshare_demo

Step 2 — Connect to BigQuery

dbt needs to know which database to talk to. You configure this in a file called profiles.yml. For BigQuery, it uses your Google Cloud credentials.

~/.dbt/profiles.yml
austin_bikeshare_demo: target: dev outputs: dev: type: bigquery method: oauth project: your-gcp-project-id dataset: austin_bikeshare_dev threads: 4 timeout_seconds: 300

Step 3 — Declare your sources

Before you can use the raw BigQuery public tables, you tell dbt where they live. This is done in a sources.yml file:

models/sources.yml
version: 2 sources: - name: austin_bikeshare database: bigquery-public-data # the Google project schema: austin_bikeshare # the dataset name tables: - name: bikeshare_trips - name: bikeshare_stations
💡 Why declare sources?
Instead of hardcoding bigquery-public-data.austin_bikeshare.bikeshare_trips in every SQL file, you give it a short alias. In your models you can then write {{ source('austin_bikeshare', 'bikeshare_trips') }} — and if the source ever moves, you change one file, not dozens.

The project folder structure

austin_bikeshare_demo/
austin_bikeshare_demo/ ├── dbt_project.yml # main config: project name, model settings ├── models/ # your transformation SQL goes here │ ├── sources.yml # where the raw data comes from │ ├── fact_trips.sql # the fact table model │ ├── schema.yml # tests and documentation │ └── star/ # dimension tables live here │ ├── dim_station.sql │ └── schema.yml └── snapshots/ # time-travel snapshots └── station_snapshot.sql

Snapshots — Time Travel for Your Data
How to track changes in source data over time

Bike stations change. A station might move to a new address, increase its number of docks, or be taken over by the city from a private operator. The raw bikeshare_stations table only shows you the current state — the history is lost.

📷 Analogy
A snapshot is like taking a daily photograph of a whiteboard. Today's photo shows today's information. Tomorrow's photo captures any changes. By keeping all the photos, you can see when things changed — and what they looked like before.

This technique is called a Type-2 Slowly Changing Dimension (SCD Type 2). Don't be put off by the name — it just means "we add new rows to record changes, instead of overwriting old ones."

How the snapshot SQL looks

snapshots/station_snapshot.sql
-- Tell dbt this is a snapshot (not a regular model) {% snapshot station_snapshot %} {{ config( target_schema='snapshots', unique_key='station_id', -- which column identifies a row strategy='check', -- "check" means: look for any changes check_cols='all' -- check ALL columns for changes ) }} SELECT * FROM {{ source('austin_bikeshare', 'bikeshare_stations') }} {% endsnapshot %}

What dbt adds automatically

When you run dbt snapshot, dbt adds two special columns to every row:

Column Added by dbtMeaning
dbt_valid_fromWhen this version of the record started being true
dbt_valid_toWhen this version stopped being true (NULL means it's still current)

Example: A station changes its dock count

station_idnamenumber_of_docksdbt_valid_fromdbt_valid_to
2538Congress Ave & 2nd St152020-01-012022-06-15
2538Congress Ave & 2nd St192022-06-15NULL (current)

↑ The station had 15 docks until June 2022, then was upgraded to 19 docks. Both records are preserved.

💡 Why does this matter for analysis?
If you're analyzing trips from 2021, you want to know that the station had 15 docks at that time — not the current 19. The snapshot lets you join trip data to the station record that was valid at the time of the trip, giving you historically accurate analysis.

dbt Models — Writing the Transformations
SQL files that define your analysis-ready tables

A dbt model is just a SQL SELECT statement saved as a .sql file. dbt reads the file, wraps it in a CREATE TABLE or CREATE VIEW statement, and runs it in BigQuery. You write the query logic; dbt handles the deployment.

The fact model selects directly from the raw source table. We pick the columns we care about and give them clear names.

models/fact_trips.sql
SELECT trip_id, subscriber_type, bike_id, start_time, start_station_id, end_station_id, duration_minutes FROM -- {{ source(...) }} tells dbt: look up this table in sources.yml {{ source('austin_bikeshare', 'bikeshare_trips') }}
💡 What dbt does with this file
dbt compiles this into:
CREATE OR REPLACE VIEW `your-project`.`dev`.`fact_trips` AS SELECT trip_id, ...
Your BigQuery project now has a clean fact_trips view!

The dimension model reads from the snapshot table (not the raw source) so it includes the historical version tracking columns.

models/star/dim_station.sql
SELECT station_id, name, location, address, property_type, number_of_docks, dbt_valid_from, -- added by dbt snapshot dbt_valid_to -- NULL means this is the current record FROM -- {{ ref(...) }} tells dbt: find the model/snapshot named 'station_snapshot' {{ ref('station_snapshot') }}
🔗 ref() vs source()
  • {{ source() }} — points to a raw external table you don't own
  • {{ ref() }} — points to another dbt model or snapshot in your project
Using ref() tells dbt about dependencies, so it knows to build station_snapshot before dim_station.

The dbt_project.yml file is the master config. Here we tell dbt to materialize our star schema models as physical tables (not views) in a dedicated star schema:

dbt_project.yml
name: 'austin_bikeshare_demo' version: '1.0.0' profile: 'austin_bikeshare_demo' models: austin_bikeshare_demo: star: # the /models/star/ subfolder +materialized: table # build as a real table, not a view +schema: star # deploy into a 'star' schema in BigQuery
💡 View vs Table
A view re-runs the SQL every time someone queries it — always fresh, but slower for big datasets. A table stores the result physically — faster queries, but you need to refresh it when source data changes.

Data Quality Tests
Trust but verify — automated checks on your data

After building your tables, how do you know the data is correct? dbt lets you define tests in YAML files that run automatically to catch data quality issues before they reach your business analysts.

🏥 Analogy
Think of dbt tests like a health check-up. You don't wait until someone feels sick to go to the doctor — you run regular tests to catch problems early. Data tests work the same way.

Tests for the fact table

models/schema.yml
version: 2 models: - name: fact_trips columns: - name: trip_id tests: - unique # every trip must have a different ID - not_null # no trip can be missing an ID - name: start_time tests: - not_null # every trip must have a start time - name: duration_minutes tests: - not_null # every trip must have a duration - name: start_station_id tests: - relationships: to: ref('dim_station') # must link to a real station field: station_id severity: warn # warn, don't fail (old stations may be missing)

Tests for the dimension table

models/star/schema.yml
version: 2 models: - name: dim_station columns: - name: station_id tests: - unique - not_null - name: name tests: - not_null - name: location tests: - not_null

Built-in dbt tests explained

Test NameWhat it checksFails when…
uniqueNo two rows have the same valueDuplicate IDs exist
not_nullNo blank/missing valuesA row has a NULL in that column
relationshipsForeign key integrityA trip references a non-existent station
accepted_valuesColumn only contains specific valuesAn unexpected value appears
⚠️ Real-World Note
Our data shows that about 110,000 trips reference station IDs that no longer exist (decommissioned stations). This is why we set severity: warn on the relationship test — it flags the issue without blocking the pipeline. In a real business scenario, you'd investigate whether to ignore, backfill, or archive this data.

Running the Pipeline
Four commands that build your data warehouse

You've written all the files. Now it's time to execute the pipeline. There are four dbt commands you'll use — in order:

  • 1
    dbt debug — Check your connection Verifies that dbt can talk to BigQuery. Run this first if something isn't working.
    terminal
    dbt debug
  • 2
    dbt snapshot — Capture the current state of stations Runs your snapshot file and creates/updates the station_snapshot table in BigQuery with SCD Type 2 history.
    terminal
    dbt snapshot

    ✓ Expected output: 1 of 1 OK snapshotted snapshots.station_snapshot

  • 3
    dbt run — Build the models Runs all your SQL models in the correct dependency order: dim_station depends on the snapshot, so dbt builds snapshot → dim_station → fact_trips.
    terminal
    dbt run

    ✓ Expected output: 2 of 2 OK created sql table/view

  • 4
    dbt test — Validate data quality Runs all tests defined in your schema.yml files. Checks for nulls, duplicates, and referential integrity.
    terminal
    dbt test

    ✓ Expected: 0 errors, 1 warning (the FK relationship warning we set intentionally)
